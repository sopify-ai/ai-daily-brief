# 封面图 + 内容升级 — 背景

## 需求来源

当前 ai-daily-brief 飞书推送为纯文本交互卡片，缺乏视觉吸引力。希望：

1. 结合 doc-to-sketch Skill 为每日简报生成一张封面图
2. 封面图能在飞书卡片中展示（尺寸合适、支持 img_key）
3. 参考微信公众号 AI 每日简报的内容深度做优化

## 参考对标：微信 AI 每日简报

微信版内容结构（远比当前丰富）：

| 板块 | 说明 | 当前是否有 |
|------|------|-----------|
| 📊 今日要闻 (Top 10) | 每条详细摘要 + 关键数据 + 来源链接 | ❌ 只有 1 句编辑观点 |
| 💡 AI 大佬观点 | 行业领袖引言 + 背景 | ❌ 只有 AI 生成金句 |
| 📊 行业数据 | 投资/收入/市场数据，结构化呈现 | ❌ 无 |
| 🏗️ 技术趋势 | 技术方向归纳（如安全攻防战） | ❌ 无 |
| 📈 企业动态 | 组织架构/产品/市场竞争 | ❌ 无 |
| ⚠️ 风险与争议 | 信任/社会影响/安全漏洞 | ❌ 无 |
| 📝 本周总结 | 周维度回顾 + 趋势矛盾分析 | ❌ 无 |

## 当前架构约束

### 飞书推送现状

- 使用 **Webhook 自定义机器人**（非自建应用）
- 卡片为纯文本 interactive card（markdown 元素）
- 卡片大小限制 ~30KB，已有截断逻辑
- feishu_push.py 完全独立，不修改 main pipeline

### 飞书图片约束

- 飞书卡片 `img` 元素只支持 `img_key`，**不支持外部 URL**
- 获取 `img_key` 需要飞书自建应用：
  1. 创建应用 → app_id + app_secret
  2. 获取 tenant_access_token
  3. POST `/open-apis/im/v1/images` 上传图片
  4. 返回 image_key
- 当前只有 Webhook 机器人，**需要新建飞书自建应用**才能实现图片上传

### doc-to-sketch 能力

- 支持 21:9 封面 (2520×1080) 和 16:9 正文 (1920×1080)
- `generate_image.sh` 可在 CI 中调用（需 IMAGE_API_KEY + IMAGE_API_URL）
- 有完整的 theme-tokens、prompt-patterns 体系
- 手绘中文技术风格，适合做日报封面

## 板块取舍说明

| 微信版板块 | 是否采纳 | 理由 |
|------------|---------|------|
| 📊 今日要闻 Top 10 | ✅ 已有（焦点 + 速览） | 当前"焦点 + 速览"等价，Top 3 深度评论升级 |
| 💡 AI 大佬观点 | ✅ 升级 → `expert_quotes` | 从 AI 生成金句升级为优先提取真人引言 |
| 📊 行业数据 | ✅ 新增 → `industry_data` | 结构化数据点是微信版最大差异化亮点 |
| 🏗️ 技术趋势 | ✅ 新增 → `tech_trends` | 多条新闻归纳为趋势，信息密度高 |
| ⚠️ 风险与争议 | ✅ 新增 → `risks` | 平衡"报喜"倾向，提供对立视角 |
| 📈 企业动态 | ❌ 不采纳 | 与"热点速览"重叠度高（企业新闻已自然归入 highlights），单独拆板块会导致内容碎片化 |
| 📝 本周总结 | 🔮 Phase 2 可选 | 需读取 archives/，逻辑复杂度高，列入后续迭代 |

## 优化目标

1. **视觉层**：每日简报带封面图 + 板块配图（doc-to-sketch 手绘风格），飞书卡片展示
2. **内容层**：摘要更深、板块更丰富（对标微信版，见上方取舍表）
3. **保持架构原则**：pipeline 零侵入、fork-friendly、渐进启用
